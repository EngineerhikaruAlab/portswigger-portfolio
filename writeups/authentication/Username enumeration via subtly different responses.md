# Lab: Username enumeration via subtly different responses

> **分類**: Authentication  
> **難易度**: Practitioner  
> **学習テーマ**: username enumeration / brute-force  
> **使用ツール**: Burp Suite（Intruder / Repeater / Comparer）, Browser DevTools

---

## 1. 概要

このLabでは、認証失敗時の**わずかなレスポンス差分**を利用して有効なユーザー名を特定し、その後にパスワードを総当たりすることで認証を突破する。

主なポイントは、画面上では同じに見えるログイン失敗メッセージでも、**サーバ側処理の違いが本文やレスポンス長の差として漏れる**場合があることを観察し、それを Burp Suite で整理して exploitation につなげる点にある。

このWriteupでは、以下を中心に整理する。

- 認証処理のどこに情報漏えいがあるか
- どの観察結果から username enumeration を疑ったか
- どのような手順で有効なユーザー名とパスワードを特定したか
- なぜその挙動が脆弱性として成立するか
- 開発者視点では何を修正すべきか

---

## 2. このLabで問われていること

このLabのゴールは、候補リストから**有効なユーザー名を列挙**し、そのユーザーに対して**パスワードを総当たり**して認証を成功させ、最終的に**アカウントページへアクセス**することである。

---

## 3. 脆弱性の要点

このLabで扱う脆弱性は、**認証失敗レスポンスの不統一による username enumeration** と、その後の **password brute-force** である。

本来、ログイン失敗時は「ユーザー名が存在しない場合」と「ユーザー名は正しいがパスワードが誤っている場合」で、**同じメッセージ、同じステータスコード、同じ外形のレスポンス**を返すのが望ましい。  
しかしこのLabでは、エラーメッセージやレスポンス長にわずかな差があり、攻撃者がその差分を手掛かりに有効なユーザー名を推測できる状態になっている。

その結果、攻撃者は以下の流れで認証を突破できる。

1. レスポンス差分から有効なユーザー名を特定する  
2. そのユーザー名に対してパスワードを総当たりする  
3. 認証成功時のレスポンス変化を根拠にログイン成功を判断する  

### 3.1 本質

この脆弱性の本質は、**認証失敗時のレスポンス統一が不十分で、ユーザー名の存在有無が外部から推測できる点**にある。

### 3.2 典型的な原因

- 認証失敗時の処理分岐ごとに、異なるテンプレートやメッセージ生成処理を使っている  
- エラーメッセージの文言、句読点、空白、HTML構造が統一されていない  
- username enumeration 後のパスワード総当たりに対して、十分な試行制限や検知がない  

---

## 4. 事前観察

まず、ログイン画面に対して無効な認証情報を送信し、失敗時のレスポンスを確認した。

### 4.1 対象機能

- エンドポイント: `/login`
- 入力項目: `username`, `password`
- 関連要素: エラーメッセージ、レスポンス本文、ステータスコード、認証後の遷移

### 4.2 初期挙動

通常の失敗時には、画面上では **`Invalid username or password.`** という一般的なエラーメッセージが表示され、レスポンスは失敗ページのまま返る。

この時点では、見た目だけを見ると「ユーザー名が違うのか、パスワードが違うのか」を隠せているように見える。

### 4.3 差分として気づいた点

ユーザー名候補を Intruder で試行し、**Grep - Extract** でエラーメッセージ部分を抽出して比較すると、**1件だけ抽出結果が微妙に異なる候補**があることに気づいた。

通常は `Invalid username or password.` と表示されるのに対し、異常系のレスポンスでは**末尾のピリオドではなく trailing space が入っていた**。  
この差は画面上では見落としやすいため、必要に応じて **Comparer** に送って差分を確認した。

---

## 5. 仮説

観察結果から、「存在しないユーザー名」と「存在するユーザー名だがパスワードが誤っている場合」で、**サーバ側の認証失敗処理が異なる分岐を通っている**と考えた。

そのため、まずは**パスワードを固定してユーザー名だけを変化**させ、レスポンス差分から valid username を特定する。  
その後に、特定した username を固定して**パスワード候補だけを総当たり**する二段階方針を取った。

---

## 6. 検証手順

### 6.1 手順1: 有効なユーザー名の特定

**目的**  
レスポンス差分から valid username を見つける。

**実施内容**  
- 無効なログインを Burp で取得し、`POST /login` を Intruder に送った  
- `username` のみを payload position に設定し、候補ユーザー名リストを投入した  
- `Settings` の **Grep - Extract** で、`Invalid username or password.` のメッセージ部分を抽出対象として追加した  

**確認ポイント**  
- 抽出されたエラーメッセージ  
- レスポンス長  
- 必要に応じて Comparer による差分比較  

**結果**  
抽出結果を並べると、1件だけ他と微妙に異なるレスポンスがあった。  
その差分は、**period ではなく trailing space** になっている点だった。

**解釈**  
この候補は、存在しないユーザー名とは異なるサーバ側処理を通っている可能性が高い。  
したがって、この username を**有効なユーザー名**と判断した。

---

### 6.2 手順2: パスワードの総当たり

**目的**  
特定した valid username に対して正しい password を見つける。

**実施内容**  
- `username` を特定済みの値に固定した  
- `password` を payload position に設定し、候補パスワードリストを投入した  
- Intruder の結果一覧で、他と異なるレスポンスを探した  

**確認ポイント**  
- ステータスコード  
- 認証後の遷移を示すレスポンスの有無  

**結果**  
1件だけ **`302 Found`** を返す候補があった。

**解釈**  
通常の失敗時レスポンスではなく、認証成功後の画面遷移が起きていると判断できるため、このパスワードが正しいと判断した。

---

## 7. 攻撃の結論

最終的には、**認証失敗レスポンスの微小差分**から valid username を特定し、その username に対してパスワードを総当たりすることで Lab を解決した。

### 最短要約

- `POST /login` を Intruder に送り、ユーザー名候補ごとの失敗メッセージ差分を抽出した  
- trailing space を含む異常レスポンスから valid username を特定した  
- その username を固定し、パスワード候補を総当たりして `302 Found` を返す認証情報を見つけた  

---

## 8. 成功条件と根拠

成功したと判断した根拠は次のとおりである。

- 有効なユーザー名候補に対してのみ、他と異なる失敗メッセージが返った  
- パスワード総当たりの結果、1件だけ `302 Found` が返った  
- 認証成功後の画面遷移が確認できた  

これらにより、単に挙動が変化しただけではなく、**正しい認証情報の組み合わせを特定できたこと**を確認できた。

---

## 9. なぜ成立したのか（原因分析）

このLabで脆弱性が成立した原因は、**認証失敗時のレスポンスが完全に統一されていなかったこと**にある。

### 9.1 認証設計上の問題

認証設計では、「ユーザー名が存在しない」「パスワードが誤っている」といった異なる失敗理由があっても、**外部から見える結果は同じ**にすべきである。

しかしこのLabでは、見た目は同じエラーメッセージでも、実際には本文末尾の空白差分が残っており、結果として username enumeration が成立していた。

### 9.2 実装上の問題

このLabでは、失敗メッセージの末尾が **period ではなく trailing space** になるという typographical difference が残っていた。  
このような**人間には見えづらい1文字差**でも、Burp Intruder や Comparer を使えば十分に利用可能である。

さらに、valid username を特定した後の password guessing に対しても、十分な防御がないため、二段階で認証突破が成立する。

---

## 10. 対策

### 10.1 実装面

- 認証失敗時の本文、ステータスコード、レスポンスサイズ、HTML構造を統一する  
- メッセージ生成処理を共通化し、分岐ごとの差分を出さない  
- 不可視文字や余分な空白も含めて、レスポンスを厳密に揃える  

### 10.2 運用・設計面

- ログイン試行回数制限  
- 遅延処理や rate limiting  
- 総当たり検知の監視とアラート  

### 10.3 このLabに即した改善案

このLabでは、まず **username enumeration を成立させているレスポンス差分を除去すること** が最優先である。  
そのうえで、たとえ valid username が推測されても password brute-force が成立しないように、試行制御を追加すべきである。

---

## 11. このLabで学んだこと

このLabを通じて、**見た目が同じに見えるエラーメッセージでも、内部的な処理差が本文の微小差分として漏れることがある**と理解できた。

また、Burp Intruder は単なる総当たりツールではなく、**Grep - Extract** による応答比較や、**Comparer** による精密比較まで含めて使うことで、観察と検証の精度を上げられると分かった。

### 今後に活かせる観点

- Authentication 系では、まず失敗レスポンスの**本文・長さ・ステータスコード**を比較する  
- 差分が曖昧なときは、Intruder の抽出機能だけでなく **Comparer** も使う  
- 攻撃手順だけでなく、「なぜその差分が出たのか」という**実装上の理由**まで考える

## 13. 補足メモ

### ハマった点

このLabは、差分が大きくないため、**Length だけ見ても何が違うのか直感的に分かりにくい**。  
そのため、本文の抽出や比較機能を使わないと見落としやすい。

### 再現時の注意点

- username enumeration フェーズと password brute-force フェーズを分けて考える  
- まずは失敗メッセージの抽出位置を正しく設定する  
- 差分が見えにくい場合は Comparer に送る  

### 追加で確認したいこと

- レスポンス時間差を利用した username enumeration との比較  
- 実サービスでどの程度の差分が情報漏えいとして成立し得るか  
- Authentication 実装で message unification をどこまで厳密に行うべきか  

---

## 14. まとめ

このLabでは、**認証失敗時のレスポンス差分による username enumeration** が成立し、その結果として password brute-force の効率が大きく上がることを確認した。

重要だったのは、失敗レスポンスが一見統一されているように見えても、**末尾の空白のようなごく小さな差分**が valid username の識別材料になる点である。  
さらに、その観察結果を Burp Intruder と Comparer で検証し、最終的に `302 Found` を根拠に認証成功を判断できた。
